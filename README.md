Let's dive right into the structure of Lex and Yacc before tackling the programs. We'll start with Lex, move to Yacc, and then break down and simplify your programs step-by-step. I'll ask guiding questions along the way to help solidify these concepts.

A Lex file (usually with a `.l` extension) acts as a scanner or tokenizer. It reads a stream of characters and groups them into meaningful chunks called "tokens".

The standard Lex program is divided into three distinct sections, separated by `%%`:

```c
%{
  // 1️⃣ DECLARATIONS
  // C code, #includes, and variable declarations go here
%}

// (Optional Lex definitions can go here)

%%
  /* 2️⃣ TRANSLATION RULES */
  /* Pattern { Action } */
%%

  // 3️⃣ AUXILIARY C FUNCTIONS
  // Main function or other helper functions go here

```

Here is what happens in each section:

* 1️⃣ **Declarations**: Everything between `%{` and `%}` is copied directly to the top of the generated C file. You use this for libraries (like `<stdio.h>`) or variables you need later.
* 2️⃣ **Translation Rules**: This is the heart of Lex. You write a Regular Expression (the pattern) on the left, and the C code to execute (the action) on the right inside `{}`. Whenever Lex sees the pattern in your input text, it runs the corresponding action.
* 3️⃣ **Auxiliary Functions**: You can define your `main()` function or other helper functions here. However, when we use Lex *with* Yacc (like in your programs), the `main()` function usually lives in the Yacc file instead.

Based on this structure, if you wanted Lex to recognize the exact word "hello" and run C code to print "Greeting detected!", in which of the three sections would you put that logic, and what roughly would it look like?

### Now, let's talk about Yacc (and why your guess was awesome)

You guessed `%token`, which is exactly what we use in **Yacc**!

Lex is just a "dumb" reader. It sees "hello" and says, "Hey, I found a 'hello'!" But it doesn't understand sentence structure. That's where Yacc comes in. Lex finds the pieces (tokens), and Yacc builds the puzzle using a Context-Free Grammar (CFG).

A Yacc program (`.y` file) uses the exact same three-part structure as Lex:

```c
%{
  // 1️⃣ C DECLARATIONS
  // #includes, C variables
%}

  // 1.5️⃣ YACC DECLARATIONS
  // This is where your `%token` goes! 
  // e.g., %token NUM IDEN

%%
  /* 2️⃣ GRAMMAR RULES (CFG) */
  /* NonTerminal : Production { C action } ; */
%%

  // 3️⃣ AUXILIARY C FUNCTIONS
  // Usually main() and yyerror() go here

```

In the grammar rules section of Yacc, you write standard CFG rules, but with C code attached to them. If you had a grammar rule `S -> A B`, in Yacc it looks like this:

```c
S : A B  { printf("Found an S!"); }
  ;

```

**How they talk to each other:**
When you run a Yacc program, it automatically calls a function named `yylex()`. This function is generated by your Lex file. Every time Yacc needs the next piece of the puzzle, it calls `yylex()`, and Lex returns a token.

Before we jump into simplifying Program 1, does this relationship make sense? Lex is the "reader" passing tokens, and Yacc is the "grammar police" putting them together. Ready for Program 1?

## Boilerplate Notes

Standard build pattern:

```bash
yacc -d yacc.y
flex lex.l
cc lex.yy.c y.tab.c -o program_name -ll
./program_name
```

If you use Bison instead of Yacc, `bison -d -y yacc.y` is the equivalent parser command.

Tool meanings:
- `bison -d` or `yacc -d` generates parser source and header files.
- `flex` generates the scanner source file.
- `cc` compiles and links the generated C files into an executable.

If a grammar uses only integer semantic values, `%union` and `%type` are not required.
